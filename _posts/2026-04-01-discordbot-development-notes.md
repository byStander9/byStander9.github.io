현재 관리 중인 `discordBot` 레포를 기준으로, Discord 봇 개발 과정에서 중요했던 부분을 정리해 두려고 한다.
이 글은 단순 소개보다 현재 코드 구조와 구현 방향을 기록해 두는 메모에 가깝다.

## 레포

- 레포 주소: <https://github.com/byStander9/discordBot>
- 설명: Discord Bot (Python)
- 주요 목적: Discord 음성 채널에서 음악 재생, 대기열 관리, 자동재생 기능 제공

## 현재 확인한 구조

레포 구조는 비교적 단순한 편이다.

```text
discordBot/
├── bot.py
├── cogs/
│   └── music.py
├── README.md
└── requirements.txt
```

핵심은 다음 두 파일에 모여 있다.

- `bot.py`: 봇 진입점
- `cogs/music.py`: 음악 관련 기능이 들어 있는 Cog

기능이 커지기 전 단계에서는 이런 구조가 관리하기 편하다.
진입점은 얇게 두고, 실제 기능은 Cog 쪽에 모으는 방식이라 역할이 비교적 명확하다.

## bot.py에서 보이는 구조

`bot.py`는 전체적으로 단순하고 깔끔한 편이다.

- `discord.ext.commands.Bot` 상속
- `message_content`, `voice_states` intent 활성화
- `setup_hook()`에서 `cogs.music` 로드
- slash command 동기화 수행
- `on_ready()`에서 상태 메시지 설정
- `.env`에서 `BOT_TOKEN` 읽어 실행

즉 현재 구조는 “최소한의 진입점 + 확장 기능은 Cog로 분리”라는 전형적인 형태를 따르고 있다.
이런 방식은 나중에 관리 기능, 유틸 기능, 관리자 명령 등을 추가할 때도 확장하기 좋다.

## music.py에서 확인한 구현 포인트

`cogs/music.py`는 생각보다 기능이 많이 들어가 있다.
파일 길이도 꽤 크고, 단순 재생 명령만 있는 수준은 아니다.

현재 코드에서 확인한 핵심 요소는 다음과 같다.

### 1. yt-dlp 기반 음원 추출

- 검색용 옵션과 재생용 옵션을 분리
- URL 입력과 검색 입력을 모두 처리
- `extract_song`, `search_youtube` 헬퍼로 역할 분리

즉 사용자가 직접 URL을 넣는 경우와 검색어로 곡을 찾는 경우를 모두 고려하고 있다.

### 2. FFmpeg 경로 탐색

코드 안에 `_find_ffmpeg()`가 별도로 있고,
Windows 경로까지 포함해서 `ffmpeg` 위치를 찾도록 되어 있다.

이 부분은 꽤 실용적이다.
단순히 “ffmpeg가 PATH에 있다고 가정”하지 않고, 실행 환경 차이를 흡수하려는 흔적이 보인다.

### 3. 데이터 구조 분리

다음과 같은 데이터 구조가 따로 정의돼 있다.

- `Song`
- `LoopMode`
- `HistoryEntry`
- `GuildState`

특히 `GuildState` 안에
- 현재 곡
- 대기열
- 반복 모드
- 자동재생 여부
- 자동재생 태그
- 아티스트 다양성 옵션
- 재생 이력
- inactivity task
- prefetch task

같은 상태가 들어가 있어서, 서버(길드) 단위 상태 관리를 염두에 두고 만든 구조라는 점이 보인다.

### 4. 자동재생 관련 로직

README 기준으로도 자동재생 기능이 꽤 많이 들어가 있고, 코드에서도 관련 상수와 상태가 보인다.

예를 들면:
- 자동재생 최소/최대 길이 제한
- 제목 기반 키워드 추출
- 유사도 비교
- 최근 재생 이력 기반 중복 방지
- 아티스트 다양성 옵션
- 프리페치 관련 상태

즉 단순히 “대기열이 비면 비슷한 노래 하나 더 틀기” 정도가 아니라,
조금 더 실제 사용성을 고려한 추천/보조 로직을 넣으려는 방향으로 보인다.

### 5. 검색 결과 선택 UI

파일 안에 `SearchSelectView`가 있고,
검색 결과를 버튼으로 고르도록 만든 구조가 확인된다.

이 부분은 Discord 봇 사용성을 꽤 크게 바꾸는 요소다.
그냥 텍스트 명령만 받는 방식보다,
검색 결과를 보여주고 버튼 선택으로 이어가는 쪽이 훨씬 Discord 인터페이스와 잘 맞는다.

## README 기준 주요 기능

README에 정리된 기능만 봐도 현재 봇은 아래 범위를 이미 다루고 있다.

- `/join`, `/leave`
- `/play`
- `/pause`, `/resume`, `/stop`
- `/skip`
- `/queue`, `/remove`
- `/loop`, `/shuffle`
- `/search`
- `/nowplaying`
- `/autoplay`

여기에 더해 다음 요소도 들어가 있다.

- 유사곡 자동재생
- 곡명 유사도 기반 중복 방지
- 아티스트 다양성 옵션
- 장르 프리셋
- 프리페치
- 음성 채널/유휴 상태 자동 퇴장

이 정도면 단순 학습용 예제를 넘어서,
직접 써보면서 기능을 점점 덧붙인 프로젝트라는 인상이 있다.

## 개발하면서 중요했던 점

Discord 음악 봇은 겉보기보다 신경 쓸 요소가 많다.

- 봇 토큰 관리
- Developer Portal 설정
- Intent 설정
- 서버 초대 권한
- 음성 채널 연결 상태
- FFmpeg 설치 여부
- yt-dlp 동작 여부
- slash command 동기화
- 길드별 상태 관리

특히 “코드는 맞는데 봇이 반응하지 않음” 또는 “메시지는 되는데 음성이 안 나옴” 같은 문제는,
코드 자체보다 환경 설정이나 권한, 외부 도구 설치에서 생기는 경우가 많다.

## 이 레포를 다시 볼 때 확인하면 좋은 것

다시 구조를 정리하거나 기능을 확장할 때는 아래 순서로 보면 좋다.

1. `bot.py`에서 진입 흐름 확인
2. `music.py` 안에서 상태 관리 구조 확인
3. 명령어 처리와 재생 로직이 어떻게 이어지는지 확인
4. 검색/선택 UI와 autoplay 로직이 어느 정도 결합되어 있는지 확인
5. 파일 분리가 더 필요한 지점이 있는지 보기

특히 `music.py`가 커질수록 다음처럼 쪼개는 것도 생각해볼 수 있다.

- `player.py`
- `queue.py`
- `autoplay.py`
- `views.py`
- `models.py`

지금은 한 파일에 모여 있어도 되지만, 기능이 더 늘어나면 유지보수성이 급격히 떨어질 수 있다.

## 앞으로 보강하면 좋을 부분

이 레포를 다음 단계로 가져가려면 아래 항목을 추가로 정리하면 좋다.

- `.env.example` 추가
- 실행 환경(Windows/Linux)별 FFmpeg 준비 방법 정리
- 에러 상황별 로그 메시지 정리
- 명령어별 예외 처리 정리
- Cog 내부 로직 파일 분리
- 실제 배포/상시 실행 방법 문서화

## 메모

지금 `discordBot`은 단순 자동응답 봇이 아니라,
실제로 음악 재생과 대기열, 검색, 자동재생, 상태 관리까지 다루는 프로젝트 형태로 가고 있다.

이런 종류의 봇은 기능 하나를 추가하는 것보다,
기능이 늘어날수록 구조를 어떻게 유지할지가 더 중요해진다.
그래서 앞으로 이 레포를 손볼 때는 “새 기능 추가”와 함께 “파일 분리와 상태 관리 구조 정리”를 같이 보는 게 중요하다고 느꼈다.
